## 1. 서비스 정의

**식식이OS는 발주 시스템이 아니다.**
식당과 공급자의 거래를 자동으로 생성하고, 돈의 흐름까지 관리하는 **외식 공급망 운영 시스템**이다.

### 핵심 가치

| 기존 관점 | 식식이OS 관점 |
|----------|-------------|
| 발주를 쉽게 | 덜 틀리게 운영하게 |
| 데이터 제공 | 행동 제시 |
| 기능 제공 | 루틴 형성 |

### 설계 철학

```
입력 줄인다  →  데이터 쌓인다
데이터 쌓인다  →  판단 자동화
판단 자동화  →  돈 번다
```

---

## 2. 사용자

### 식당 (구매자)
- 식자재를 발주하고 구매하는 주체
- 식당OS를 통해 발주요청 생성, 공급자 선택, 결제 관리
- 입력 최소화 / 추천 중심 UX

### 공급자 (판매자)
- 식자재를 공급하는 주체
- 공급자OS를 통해 발주요청 수신, 입찰, 주문 처리, 수금 관리
- ERP 기능 + 플랫폼을 통한 영업 기회

### 플랫폼 관리자
- 전체 시스템을 운영·조정·실험하는 컨트롤 타워
- 관리자OS를 통해 거래 관제, 수치/문구 제어, 정산 관리

---

## 3. 핵심 흐름

### 거래 흐름 (식당 → 공급자)

```
[1] 식당: 발주요청(RFQ) 생성
      ↓
[2] 기존 거래처 우선 공개
    → 일정 시간 후 확장 공개
      ↓
[3] 공급자: 입찰 / 조건 제안
      ↓
[4] 식당: 선택 / 협상 수락
      ↓
[5] 주문 확정
      ↓
[6] 납품 완료
      ↓
[7] 매출 / 매입 / 원장 자동 생성
```

### 돈 흐름 원칙

```
모든 데이터는 거래에서 생성된다
```

- 매출 — 납품 기준으로 자동 계산
- 매입 — 주문 기준으로 자동 생성
- 원장 — 거래 기준으로 자동 생성
- 정산 — 플랫폼이 관리

## 4. 주요 기능

### 4-1. 발주요청 (RFQ)

**입력 구조**

| 구분 | 항목 |
|------|------|
| 필수 | 제품 (SKU 기준), 수량 |
| 조건부 필수 | 현재 구매 가격 (기존 제품일 경우) |
| 선택 | 납기, 요청 조건 |
| 자동 | 위치, 과거 가격, 거래 이력, 공급자 |

**상태 흐름**

```
draft → open → bidding → counter_offered → selected → ordered → completed
                    ↓
                 expired
```

**분기 조건**

```
기존 제품 → 현재 구매 가격 입력 필수
신규 제품 → 공급자 제안 방식
```

**협상 기능**
- 공급자가 조건 제안 가능 (수량 변경, 가격 변경)
- 식당 수락 시 조건 자동 변경

### 4-5. 공통 UX 원칙 ← [추가]

**행동 선택 원칙**
- 막지 않는다 → 선택하게 만든다

**협상 UX 원칙** ← [추가]
- 협상 시 시스템은 거래 성사 가능성과 수익 기회를 함께 안내한다
- 사용자의 행동을 제한하지 않고 선택을 돕는 구조로 설계한다

**실패 / 로딩 UX 원칙** ← [추가]
- 핵심 액션(발주요청, 입찰, 주문 확정, 결제 등)은 실패 처리, 로딩 상태, 취소/되돌리기 기능을 포함한다
- 시스템은 항상 현재 상태를 명확히 표시한다
- 사용자가 다음 행동을 이해할 수 있도록 안내한다

---

### 4-2. 자동화 영업 (핵심 수익 엔진)

거래처 데이터를 기반으로, 적절한 타이밍에, 자동으로 거래를 생성하는 시스템.

**구조**
```
타겟 + 트리거 + 메시지 + 채널 + 성과 분석
```

**트리거 (7개 고정)**

1. 구매 후 n일
2. 미구매 n일
3. 재구매 타이밍
4. 가격 변화
5. 신규 고객
6. 주문 감소
7. 견적 / 장바구니 이탈

**채널**
- 문자 (단문 중심)
- 카카오톡 (전환 중심)
- 이메일 (제안 / 대량)

**문자 규격**
- 90 bytes 이하 → SMS
- 초과 → LMS
- 실시간 바이트 표시 필수

**스크립트 관리**
- 내부 저장 (구글시트 사용 금지)
- 버전 관리
- 트리거 연결
- 채널 분기

**AI 역할**
- 메시지 추천
- 타이밍 추천
- 채널 추천
- "생성"이 아니라 "행동 제안"

---

### 4-3. 매입 처리 (공급자)

```
재고 없음  →  주문 = 자동 매입

재고 있음  →  재고 차감 후 부족분만 매입
```

---

### 4-4. 원장

- 자동 생성 (거래 기준)
- 수동 입력 없음

## 5. 앱별 구조

### 식당OS

```
오늘        — 당일 할 일 중심 대시보드
발주        — 발주요청 생성 / 관리
돈          — 지급 예정 / 결제 현황
거래처      — 공급자 목록 / 관리
상품        — 식자재 + 메뉴 연결 / 원가·마진 자동 계산
설정        — 식당 정보 / 고정비
```

**상품 입력 방식**
- 사진 인식
- 바코드 인식
- 검색
- "입력"이 아니라 "인식"

---

### 공급자OS

```
발주요청    ★ (플랫폼 핵심)
대시보드    — 주요 지표
거래처관리  — CRM (액션 스코어 기반)
주문관리    — 주문 접수 / 처리 / 납품
상품관리    — 상품 등록 / 단가 관리
매입관리    — 자동 매입 처리
원장관리    — 자동 원장
매출분석    — 매출 집계 / 분석
자동화영업  — 거래처 대상 자동 영업
```

---

### 관리자OS

```
통합 대시보드   — 전체 현황
거래 관제       — 전체 거래 모니터링
거래처 관리     — 식당 + 공급자 + 일반
데이터 조회     — 식자재 / 메뉴 / 가격
자동화영업      — 전체 DB 대상 직접 영업
돈 흐름         — 전체 거래 / 정산 / 미수금
설정 / 실험     — 수치·문구 제어
```

**관리자 핵심 권한**

1. 수치 제어 — 트리거 기준, 발주 공개 시간 (코드 수정 없이 변경)
2. 문구 제어 — 메시지, 버튼, 경고 텍스트 (코드 수정 없이 변경)
3. 데이터 조회 — 식당 메뉴, 식자재, 가격 전체
4. 자동화영업 — 전체 DB 대상 직접 영업 실행
5. 돈 흐름 — 전체 거래, 정산, 미수금 관리

## 10. 사업자 구조 ← [추가]

### 법인 분리

| 사업자 | 역할 | 운영 앱 |
|--------|------|---------|
| 디닷페이스 | 플랫폼 운영사 | 관리자OS |
| 해내음코리아 | 공급자 (초기 유일한 공급자) | 공급자OS |

### 돈의 흐름

```
식당 → 디닷페이스 입금 (계산서 발행)
        ↓
디닷페이스 → 공급자 정산 (공급자가 계산서 발행)
```

- 디닷페이스가 돈의 흐름 장악
- 수수료/정산은 관리자OS에서 관리
- 식식이몰 매출 사업자: 해내음코리아 (별도 정산 구조 필요)

### 초기 운영 전략

```
1단계: 식당 모으기 우선
       → 공급자 없을 때 디닷페이스/해내음코리아가 직접 처리
       → 관리자OS에서 발주요청 확인 후 공급자OS로 배정

2단계: 공급자 회원 확보
       → 공급자OS로 직접 배정

3단계: 플랫폼 완전 자동화
       → 관리자OS는 관제만
```

## 11. 고객 분류 관리 ← [추가]

### 설계 원칙

> 분류는 수시로 바뀐다. 코드에 하드코딩하지 않는다.
> 설정 페이지에서 정무님이 직접 추가/수정/삭제할 수 있어야 한다.

### 분류 체계

| 분류 종류 | 설명 | 관리 방식 |
|----------|------|----------|
| 구매채널 | 쿠팡/스마트스토어(A)/스마트스토어(B)/식식이몰/오프라인 등 | 자유 추가/수정/삭제 |
| 식식이 회원 등급 | 일반회원/사업자회원/구독회원/협력업체 등 | 자유 추가/수정/삭제 |
| 식식이OS 구독 여부 | 발주요청으로만 주문하는 고객 구분 | Y/N |
| 영업 관리 방식 | 자동/수동/제외 | 고정 |
| 연락 선호 채널 | 문자/카카오/전화/방문 (복수 선택) | 고정 |
| 수신 동의 | 마케팅 수신 동의 여부 | Y/N (법적 의무) |
| 안심번호 여부 | 쿠팡 등 안심번호 사용 여부 | Y/N |

### 설정 메뉴 위치

```
설정
ㄴ 테넌트 설정
ㄴ 메시지 템플릿
ㄴ 고객 분류 관리  ← 분류 종류 + 항목 모두 추가/수정/삭제
ㄴ 영업 채널 설정  ← 알리고 API, 카카오 비즈메시지 설정
```

### 거래처 등록 폼 추가 필드

```
기존:
- 고객유형 (사업자/개인/예비)
- 사업자등록번호, 상호명, 대표자명, 업태
- 연락처, 주소, 결제조건, 목표월매출, 역할, 유입경로

추가:
- 이메일
- 네이버플레이스 URL (입력 시 자동 분석)
- 구매채널 (설정에서 관리)
- 식식이 회원 등급 (설정에서 관리)
- 식식이OS 구독 여부
- 연락 선호 채널 (복수 선택)
- 영업 관리 방식 (자동/수동/제외)
- 수신 동의 여부
- 안심번호 여부
```

## 12. 자동화영업 상세 ← [전면 보완]

### 메뉴 구조

```
자동화영업
ㄴ 영업스케줄 (캘린더)
ㄴ 영업이력
ㄴ 스크립트관리
```

### 영업스케줄

```
- 캘린더 UI (월/주/일 단위 전환)
- 수동 입력: 정무님이 직접 스케줄 등록
  예) "A거래처 3일 뒤 C식자재 제안 연락"
- 자동 입력: 수금 알림, 문자 발송 기록, 재구매 타이밍 등
- AI 학습: 거래처별 최적 타이밍/방식 자동 추천
```

### 영업이력

```
기록 항목:
- 연락 날짜/시간
- 연락 방식 (문자/전화/카카오/방문)
- 연락 후 메모
- 실제 주문 연결 (주문등록과 자동 연동)

AI 학습 데이터:
- 연락 날짜/시간/방식
- 연락 후 주문 여부 및 금액
- 메모 내용
→ 거래처별 최적 패턴 도출
→ 예) A거래처는 수요일 오후 4시 문자 효과 좋음
```

### 전화 기능

- 거래처 전화번호 클릭 시 바로 전화 연결 (tel: 링크)
- 전화 후 영업이력 자동 기록 유도

### 네이버플레이스 자동 분석

```
URL 입력 시:
- 메뉴 자동 파악
- 후기 분석 (긍정/부정)
- 업소 유형 파악
- 연관 식자재 자동 추천
- 분석 방식: OpenAI Vision (스크린샷 기반)
- 크롤링 금지 (네이버 정책)
```

### 자동화영업 트리거 우선순위

```
1순위 — 타 채널 구매 고객 → 식식이몰 유도
  - 쿠팡/스마트스토어 안심번호: 주문 후 하루 이내 발송 필수
  - 안심번호 → 단문(SMS 90바이트)만 발송
  - 문자 내용: 수령 확인 + 식식이몰 안내 + 가격 혜택

2순위 — 식식이몰 회원 → 객단가 높이기
```
