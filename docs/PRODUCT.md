## 1. 서비스 정의

**식식이OS는 발주 시스템이 아니다.**
식당과 공급자의 거래를 자동으로 생성하고, 돈의 흐름까지 관리하는 **외식 공급망 운영 시스템**이다.

### 핵심 가치

| 기존 관점 | 식식이OS 관점 |
|----------|-------------|
| 발주를 쉽게 | 덜 틀리게 운영하게 |
| 데이터 제공 | 행동 제시 |
| 기능 제공 | 루틴 형성 |

### 설계 철학

```
입력 줄인다  →  데이터 쌓인다
데이터 쌓인다  →  판단 자동화
판단 자동화  →  돈 번다
```

---

## 2. 사용자

### 식당 (구매자)
- 식자재를 발주하고 구매하는 주체
- 식당OS를 통해 발주요청 생성, 공급자 선택, 결제 관리
- 입력 최소화 / 추천 중심 UX

### 공급자 (판매자)
- 식자재를 공급하는 주체
- 공급자OS를 통해 발주요청 수신, 입찰, 주문 처리, 수금 관리
- ERP 기능 + 플랫폼을 통한 영업 기회

### 플랫폼 관리자
- 전체 시스템을 운영·조정·실험하는 컨트롤 타워
- 관리자OS를 통해 거래 관제, 수치/문구 제어, 정산 관리

---

## 3. 핵심 흐름

### 거래 흐름 (식당 → 공급자)

```
[1] 식당: 발주요청(RFQ) 생성
      ↓
[2] 기존 거래처 우선 공개
    → 일정 시간 후 확장 공개
      ↓
[3] 공급자: 입찰 / 조건 제안
      ↓
[4] 식당: 선택 / 협상 수락
      ↓
[5] 주문 확정
      ↓
[6] 납품 완료
      ↓
[7] 매출 / 매입 / 원장 자동 생성
```

### 돈 흐름 원칙

```
모든 데이터는 거래에서 생성된다
```

- 매출 — 납품 기준으로 자동 계산
- 매입 — 주문 기준으로 자동 생성
- 원장 — 거래 기준으로 자동 생성
- 정산 — 플랫폼이 관리

## 4. 주요 기능

### 4-1. 발주요청 (RFQ)

**입력 구조**

| 구분 | 항목 |
|------|------|
| 필수 | 제품 (SKU 기준), 수량 |
| 조건부 필수 | 현재 구매 가격 (기존 제품일 경우) |
| 선택 | 납기, 요청 조건 |
| 자동 | 위치, 과거 가격, 거래 이력, 공급자 |

**상태 흐름**

```
draft → open → bidding → counter_offered → selected → ordered → completed
                    ↓
                 expired
```

**분기 조건**

```
기존 제품 → 현재 구매 가격 입력 필수
신규 제품 → 공급자 제안 방식
```

**협상 기능**
- 공급자가 조건 제안 가능 (수량 변경, 가격 변경)
- 식당 수락 시 조건 자동 변경

### 4-5. 공통 UX 원칙 ← [추가]

**행동 선택 원칙**
- 막지 않는다 → 선택하게 만든다

**협상 UX 원칙** ← [추가]
- 협상 시 시스템은 거래 성사 가능성과 수익 기회를 함께 안내한다
- 사용자의 행동을 제한하지 않고 선택을 돕는 구조로 설계한다

**실패 / 로딩 UX 원칙** ← [추가]
- 핵심 액션(발주요청, 입찰, 주문 확정, 결제 등)은 실패 처리, 로딩 상태, 취소/되돌리기 기능을 포함한다
- 시스템은 항상 현재 상태를 명확히 표시한다
- 사용자가 다음 행동을 이해할 수 있도록 안내한다

---

### 4-2. 자동화 영업 (핵심 수익 엔진)

거래처 데이터를 기반으로, 적절한 타이밍에, 자동으로 거래를 생성하는 시스템.

**구조**
```
타겟 + 트리거 + 메시지 + 채널 + 성과 분석
```

**트리거 (7개 고정)**

1. 구매 후 n일
2. 미구매 n일
3. 재구매 타이밍
4. 가격 변화
5. 신규 고객
6. 주문 감소
7. 견적 / 장바구니 이탈

**채널**
- 문자 (단문 중심)
- 카카오톡 (전환 중심)
- 이메일 (제안 / 대량)

**문자 규격**
- 90 bytes 이하 → SMS
- 초과 → LMS
- 실시간 바이트 표시 필수

**스크립트 관리**
- 내부 저장 (구글시트 사용 금지)
- 버전 관리
- 트리거 연결
- 채널 분기

**AI 역할**
- 메시지 추천
- 타이밍 추천
- 채널 추천
- "생성"이 아니라 "행동 제안"

---

### 4-3. 매입 처리 (공급자)

```
재고 없음  →  주문 = 자동 매입

재고 있음  →  재고 차감 후 부족분만 매입
```

---

### 4-4. 원장

- 자동 생성 (거래 기준)
- 수동 입력 없음

## 5. 앱별 구조

### 식당OS

```
오늘        — 당일 할 일 중심 대시보드
발주        — 발주요청 생성 / 관리
돈          — 지급 예정 / 결제 현황
거래처      — 공급자 목록 / 관리
상품        — 식자재 + 메뉴 연결 / 원가·마진 자동 계산
설정        — 식당 정보 / 고정비
```

**상품 입력 방식**
- 사진 인식
- 바코드 인식
- 검색
- "입력"이 아니라 "인식"

---

### 공급자OS

```
발주요청    ★ (플랫폼 핵심)
대시보드    — 주요 지표
거래처관리  — CRM (액션 스코어 기반)
주문관리    — 주문 접수 / 처리 / 납품
상품관리    — 상품 등록 / 단가 관리
매입관리    — 자동 매입 처리
원장관리    — 자동 원장
매출분석    — 매출 집계 / 분석
자동화영업  — 거래처 대상 자동 영업
```

---

### 관리자OS

```
통합 대시보드   — 전체 현황
거래 관제       — 전체 거래 모니터링
거래처 관리     — 식당 + 공급자 + 일반
데이터 조회     — 식자재 / 메뉴 / 가격
자동화영업      — 전체 DB 대상 직접 영업
돈 흐름         — 전체 거래 / 정산 / 미수금
설정 / 실험     — 수치·문구 제어
```

**관리자 핵심 권한**

1. 수치 제어 — 트리거 기준, 발주 공개 시간 (코드 수정 없이 변경)
2. 문구 제어 — 메시지, 버튼, 경고 텍스트 (코드 수정 없이 변경)
3. 데이터 조회 — 식당 메뉴, 식자재, 가격 전체
4. 자동화영업 — 전체 DB 대상 직접 영업 실행
5. 돈 흐름 — 전체 거래, 정산, 미수금 관리

## 9. 실행 원칙 (Agent Loop) ← [추가]

식식이OS는 문서 기반으로 실행된다.

**문서 구조**

| 문서 | 역할 |
|------|------|
| `PRODUCT.md` | 제품 정의 — 무엇을 할 것인가 |
| `CONTEXT.md` | 시스템 구조 — 어떻게 구성되어 있는가 |
| `tasks.md` | 실행 작업 — 무엇을 수정할 것인가 |

**AI 실행 범위**
- 위 문서를 기반으로 기능을 구현한다
- 코드를 수정한다
- 테스트 시나리오를 생성한다

**사람 역할**
- 최종 검증과 운영 판단은 사람이 수행한다

모든 개발은 문서를 기반으로 자동화된다.
